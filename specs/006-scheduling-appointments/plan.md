# Implementation Plan: Scheduling & Appointments

**Branch**: `006-scheduling-appointments` | **Date**: 2026-04-12 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-scheduling-appointments/spec.md`

## Summary

Implement complete appointment scheduling with booking, daily views, provider availability, status management, and rescheduling. Uses Supabase PostgreSQL with EXCLUDE constraint for double-booking prevention, time slot computation from provider availability minus existing appointments, and a multi-step booking flow. Integrates with 005-patients for patient search/selection and 004-clinic-tenancy for provider/staff context.

## Design Decisions

### 1. Available Slot Computation Algorithm

The slot computation runs **client-side** from two server queries:

```
GetAvailableSlots(providerId, date, clinicId, appointmentTypeId):
  1. Query provider_availability WHERE provider_id = providerId
     AND day_of_week = date.weekday AND is_active = true
  2. Query provider_blocked_dates WHERE provider_id = providerId
     AND blocked_date = date
     → If any blocked date exists, return empty list
  3. Query appointments WHERE provider_id = providerId
     AND start_time::date = date
     AND status NOT IN ('cancelled', 'no_show')
  4. Get appointment_type duration from appointmentTypeId
  5. For each availability window [start_time, end_time]:
     a. Generate candidate slots at `duration_minutes` intervals
        starting from window start_time
     b. For each candidate slot [slot_start, slot_start + duration]:
        - Check no overlap with any existing appointment
        - Check slot_start + duration ≤ window end_time
        - If valid, add to available_slots list
  6. Return available_slots sorted by start_time
```

This approach is simple, correct, and efficient for typical clinic volumes (~20 appointments/provider/day). The EXCLUDE constraint is the safety net — even if the client-side computation has a race condition, the database prevents the double-booking.

### 2. Appointment Status State Machine

```
                    ┌──────────┐
                    │ scheduled │
                    └─────┬────┘
                          │
            ┌─────────────┼─────────────┐
            │             │             │
            ▼             ▼             ▼
      ┌───────────┐ ┌─────────┐  ┌──────────┐
      │ confirmed │ │cancelled│  │ no_show  │
      └─────┬─────┘ └─────────┘  └──────────┘
            │
            ▼
      ┌────────────┐
      │ checked_in │──────────────► cancelled
      └─────┬──────┘
            │
            ▼
      ┌─────────────┐
      │ in_progress │─────────────► cancelled
      └──────┬──────┘
             │
             ▼
       ┌───────────┐
       │ completed │  (terminal — no transitions out)
       └───────────┘
```

**Valid transitions map** (implemented in domain entity):
```dart
static const Map<AppointmentStatus, Set<AppointmentStatus>> _validTransitions = {
  scheduled:  {confirmed, cancelled, noShow},
  confirmed:  {checkedIn, cancelled},
  checkedIn:  {inProgress, cancelled},
  inProgress: {completed, cancelled},
  completed:  {},        // terminal
  cancelled:  {},        // terminal
  noShow:     {},        // terminal
};
```

The `canTransitionTo(newStatus)` method validates client-side. The database column is a text enum — no DB-level state machine constraint (kept simple for v1).

### 3. Multi-Step Booking Flow Architecture

The `BookAppointmentCubit` manages a linear step flow:

```
Step 0: Select Patient → uses SearchPatients use case from 005-patients
Step 1: Select Appointment Type → uses GetAppointmentTypes use case
Step 2: Select Provider + Date + Time Slot → uses GetAvailableSlots
Step 3: Confirm & Book → uses CreateAppointment use case
```

State tracks:
- `currentStep` (0-3)
- `selectedPatient` (Patient entity from 005-patients)
- `selectedAppointmentType` (AppointmentType entity)
- `selectedProvider` (StaffAssignment from clinic context)
- `selectedDate` (DateTime)
- `selectedSlot` (TimeSlot)
- `availableSlots` (List<TimeSlot>)
- `isLoading`, `error`

For **rescheduling**, the cubit is initialized with patient and provider pre-selected, starting at Step 2 (slot selection). On confirm, it cancels the old appointment (reason: "Rescheduled") and creates the new one.

### 4. Integration with Patient Search (005-patients)

The booking flow reuses the `SearchPatients` use case from the patient feature. This is injected via GetIt into `BookAppointmentCubit`:

```dart
sl.registerFactory(
  () => BookAppointmentCubit(
    createAppointment: sl(),
    cancelAppointment: sl(),
    getAvailableSlots: sl(),
    getAppointmentTypes: sl(),
    searchPatients: sl(),  // from patient feature DI
  ),
);
```

This follows Constitution Principle IV — no direct import of patient feature implementation. The `SearchPatients` use case and `Patient` entity are accessed through the DI container (use case registered by `initPatient()` which runs before `initScheduling()`).

### 5. Provider Identification for "My Day"

The logged-in user's staff assignment is available from `ClinicCubit.state` (which holds `List<StaffAssignment>` including the current user's assignment with their `userId` and `role`). The `MyDayCubit` receives the provider's `userId` from the auth context and queries appointments where `provider_id` matches.

Note: In the appointments table, `provider_id` references `staff_assignments.user_id` (the Supabase Auth user UUID), not the `staff_assignments.id`. This allows direct filtering by the logged-in user's auth ID.

### 6. Break Periods Modeling

Breaks are modeled as **gaps between consecutive availability entries** for the same day. Example:

- Entry 1: Monday 09:00–13:00 (morning session)
- Entry 2: Monday 14:00–17:00 (afternoon session)
- Implied break: 13:00–14:00

This eliminates the need for a separate `breaks` table. The slot computation algorithm iterates over each availability window independently, so breaks are naturally excluded.

### 7. Route Registration

New routes added to `app_router.dart` and `route_names.dart`:

```
/home/appointments           → AppointmentListPage (daily view with filters)
/home/appointments/book      → BookAppointmentPage (multi-step booking)
/home/appointments/:id       → AppointmentDetailPage (view + status actions)
/home/my-day                 → MyDayPage (doctor's today view)
/home/availability           → ManageAvailabilityPage
/home/appointment-types      → AppointmentTypeListPage
/home/appointment-types/new  → AppointmentTypeFormPage
/home/appointment-types/:id  → AppointmentTypeFormPage (edit mode)
```

All scheduling routes are nested under the `/home` branch of the StatefulShellRoute, consistent with the existing pattern.
